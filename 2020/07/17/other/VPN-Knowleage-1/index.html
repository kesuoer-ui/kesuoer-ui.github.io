Clash、SSR、V2RayN 与代理网络原理整理

发表于2020-07-18|更新于2026-05-11|技术
|浏览量:

背景

最开始接触代理，并不是出于“研究网络”。

原因很直接：

有些网站打不开。

当时的理解非常粗糙，甚至可以总结成一句话：

能用就行。

后来接触得多了，才慢慢意识到，这套东西远不只是“能不能访问网站”。

往底层拆，其实是一整套网络流量体系：

  • TCP / UDP
  • DNS 解析
  • 路由转发
  • NAT
  • 隧道协议
  • 加密传输
  • 流量分流

再往后甚至会接触到：

  • SD-WAN
  • 零信任网络
  • 服务网格
  • 企业级 VPN 架构

当时并没有这些概念，但隐约能感觉到：

这不是“工具”，而是“网络层能力”。

最初的误解

刚开始的理解很简单：

Clash = 魔法开关

开了就通，不开就不通。

后来才发现完全不是这么回事。

Clash 本身并不“负责翻越什么”，它更像：

一个流量调度器。

真正执行网络连接的是底层协议，例如：

  • Shadowsocks
  • VMess / VLESS
  • Trojan
  • SOCKS5
  • WireGuard
  • Hysteria

Clash 的作用是：

决定“哪一类流量走哪一条通道”。

代理的本质

代理可以用一句很朴素的话描述：

让一个中间节点替你访问网络。

原始结构：

id
客户端 -> 目标网站

代理结构：

id
客户端 -> 代理服务器 -> 目标网站

目标网站看到的是：

代理服务器的 IP，而不是你的真实 IP。

HTTP 代理（早期形态）

最早接触的是 HTTP 代理。

结构很直白：

id
浏览器 -> HTTP Proxy -> 网站

特点：

  • 仅支持 HTTP/HTTPS
  • 配置简单
  • 依赖浏览器/应用支持

局限也很明显：

  • 不支持所有协议
  • UDP 基本不可用
  • 兼容性差

后面逐渐被 SOCKS5 替代。

SOCKS5：通用代理基础

SOCKS5 是后来真正用得最多的底层能力。

结构：

id
应用 -> SOCKS5 -> 代理服务器 -> 目标

特点很直接：

  • 不关心上层协议（HTTP / TCP / UDP）
  • 几乎所有应用都能用
  • 更接近“通用转发层”

很多工具本质都是在本地暴露：

127.0.0.1:7890

然后所有流量走这个入口。

SSR：早期常见方案

SSR 是早期比较常见的一种实现。

当时的特点：

  • 配置简单
  • 订阅形式传播
  • 客户端生态丰富

常见使用方式：

id
SSR 订阅链接 -> 导入客户端 -> 直接使用

但后来逐渐被替代，原因主要是：

  • 协议相对固定
  • 特征明显
  • 扩展能力有限

V2RayN：协议认知转折点

在使用 V2RayN 之后，认知发生变化。

之前的理解是：

节点就是一个“可用线路”。

之后才意识到：

节点背后是协议体系：

  • VMess
  • VLESS
  • Trojan
  • Shadowsocks
  • SOCKS

本质是：

不同的加密与隧道实现方式。

Clash：从“能用”到“可控”

Clash 真正改变的点不在于“连接能力”，而在于：

流量控制能力。

它引入了规则分流体系。

典型结构：

id
rules:
  - DOMAIN-SUFFIX,google.com,Proxy
  - DOMAIN-SUFFIX,bilibili.com,DIRECT
  - GEOIP,CN,DIRECT
  - MATCH,Proxy

含义是：

不同流量走不同出口。

分流的重要性

如果只有“全局代理”，会出现明显问题：

  • 国内网站变慢
  • 视频 CDN 绕路
  • 游戏延迟增加
  • 银行/风控异常

因此核心需求变成：

不是“能不能代理”，而是“怎么分流”。

DNS：容易被忽略的核心问题

DNS 的问题在早期并不显眼。

基本认知是：

DNS = 域名解析

但实际情况复杂得多。

DNS 泄漏问题

例如访问：

id
google.com

如果 DNS 没有走代理：

  • 请求仍然走本地运营商
  • 运营商可以知道访问目标

这就是：

DNS 泄漏

即使实际流量被代理，也仍然暴露访问意图。

Clash 的 DNS 接管

Clash 后来引入了 DNS 接管机制：

id
dns:
  enable: true
  listen: 0.0.0.0:1053

作用是：

统一控制域名解析路径。

Fake-IP 模式

第一次接触 Fake-IP 时基本是“看不懂状态”。

配置类似：

id
enhanced-mode: fake-ip

核心机制：

  • 伪造 IP 映射
  • 域名与 IP 建立内部关系
  • 强化分流控制能力

本质是：

用“虚拟 IP”控制流量路径。

TUN 模式：系统级代理

TUN 是理解代理能力的一个分界点。

在没有 TUN 时：

  • 需要应用支持代理
  • 不支持的软件无法接管

开启 TUN 后：

系统流量被整体接管。

TUN 的本质

TUN 是虚拟网卡：

id
系统 -> 虚拟网卡 -> 代理内核

所有流量包括：

  • 浏览器
  • 应用
  • 游戏
  • UDP

都会进入代理路径。

节点的真实含义

当时对“节点”的理解比较简单：

一个 IP

实际结构更复杂：

  • 地址 + 端口
  • 协议类型
  • 加密方式
  • TLS 配置
  • 路由参数

本质是：

一个完整的隧道入口配置。

代理链

后期逐渐接触到多跳结构：

id
客户端 -> 日本节点 -> 美国节点 -> 目标网站

用途主要是：

  • 多层转发
  • 隐藏真实出口
  • 特殊网络环境绕行

认知变化

早期理解：

代理 = 翻墙工具

后期理解：

代理 = 网络流量调度系统

很多技术后来才发现底层一致：

  • VPN
  • 企业内网
  • SD-WAN
  • 游戏加速器
  • 云办公网络

本质都是：

控制流量路径。

Clash 配置的工程化意义

第一次看到完整配置时：

proxies:
proxy-groups:
rules:
dns:
tun:

感受到的不是“功能”，而是：

网络系统工程结构。

回头看

SSR、V2RayN、Clash 这些工具真正带来的变化并不是“访问能力”。

而是：

第一次开始理解网络流量是如何被构建、转发和控制的。

后续无论是：

  • 服务器部署
  • Docker 网络
  • 反向代理
  • 内网穿透
  • API 转发

都能在这套理解上继续扩展。
