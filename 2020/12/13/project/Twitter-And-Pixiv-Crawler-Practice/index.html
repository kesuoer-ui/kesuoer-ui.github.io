Twitter 与 Pixiv 爬虫实践踩坑记录

这是我真正开始深入理解：

现代网站不是“网页”
而是“前端应用”

的一次项目实践。

当时主要目的其实很简单：

自动收集同人图

尤其：

  • Twitter
  • Pixiv

这种二次元内容站。

最开始的想法其实很幼稚

一开始我以为：

requests.get()

就结束了。

后来才发现：

根本不是这么回事。

当时 Twitter 还叫 Twitter

那时候：

还没改名 X。

而且：

API 也没现在这么离谱。

但：

反爬已经很严重了。

第一次真正理解“动态网页”

最开始。

我写的：

其实是：

Selenium + XPath

传统浏览器爬虫。

当时的核心思路

1. 浏览器打开页面

self.bro.get(self.link)

2. 下滑页面

window.scrollTo()

3. 获取 HTML

content = self.bro.page_source

4. XPath 提取图片

tree.xpath('//img/@src')

5. 下载图片

urllib.request.urlretrieve()

现在看起来很土

但：

当时其实已经能跑起来了。

我第一次意识到 Twitter 是“无限流”

因为：

Twitter：

根本不是：

翻页网站

而是：

无限下滑加载

所以：

传统：

下一页

逻辑直接失效。

当时的解决方案

是：

window.scrollTo()

不断下滑。

这里有个经典问题

很多新人第一次都会踩：

滚太快

因为：

页面：

还没加载完成。

你已经：

继续往下滚了

所以后来我写了：

sleep(self.waittime)

强制等待。

现在看其实不优雅

但：

早期 Selenium 项目。

几乎：

全是 sleep()

XPath 是我第一次真正理解网页结构

当时：

Chrome F12。

基本天天开着。

我那时候开始疯狂看

例如：

<div>
  <section>
    <article>
      <img />
    </article>
  </section>
</div>

然后：

XPath：

//body//main//section//article//img/@src

开始越来越长。

后来才发现

这种写法：

其实非常脆弱。

Twitter 最大问题

是：

前端结构经常改

今天还能用。

明天：

XPath 就炸了。

所以后来我越来越讨厌

绝对XPath

这是非常重要的认知变化

真正稳定的爬虫。

不能：

依赖复杂DOM结构

Twitter 图片链接其实藏了很多信息

当时我还专门处理了：

name=4096x4096

因为：

Twitter 默认给的：

往往是：

缩略图

而真正原图：

需要自己拼参数。

所以后来代码里有：

src_result = src_target[0] + '4096x4096'

这其实已经属于“接口规律分析”

了。

我第一次意识到：

很多网站资源。

其实：

URL 是有规律的

文件去重问题

当时还做了：

resource_list

去重。

因为 Twitter 无限流。

会重复加载。

如果不去重。

会疯狂下载重复图片。

文件名合法化也是经典坑

Windows：

不允许：

\/:*?"<>|

所以后来我专门写了：

replace('/', '-')

这种处理。

这是我第一次意识到

真正工程项目里。

很多时间：

根本不是写核心逻辑。

而是：

处理边角问题

后来开始尝试“用户媒体页”

再后来。

我已经不满足于：

关键词搜索

了。

因为：

搜索页：

问题太多。

例如：

  • 混杂
  • 不稳定
  • 容易漏图

所以后来转向：

用户主页/media

即：

https://twitter.com/xxx/media

这里是一个巨大认知升级

因为：

我第一次开始：

分析网站URL规律

后来很多爬虫

其实核心能力就是：

找规律。

登录是我踩过最大的坑之一

后来 Twitter：

开始限制游客访问。

于是：

必须登录。

当时我的做法

其实非常原始。

Selenium 自动输入账号密码

send_keys()

然后：

Keys.ENTER

模拟登录。

当时最麻烦的问题

是：

登录流程经常变

有时候：

是：

账号 -> 密码

有时候：

还会：

  • 邮箱验证
  • 手机验证
  • 二次确认

导致 XPath 全部失效

更痛苦的是

Twitter：

会检测：

自动化浏览器

于是：

开始出现：

  • 风控
  • 验证码
  • 登录异常

这是我第一次真正接触：

反爬

那时候第一次意识到

现代互联网。

其实：

默认不欢迎爬虫

Selenium 当年的最大问题

就是：

太重

一个浏览器。

内存：

几百MB。

多开后：

电脑直接爆炸。

所以后来：

无头模式：

--headless

几乎是必备。

但无头模式又有新问题

很多网站：

会检测：

Headless Browser

所以后来：

反而：

有头浏览器更稳定

EdgeDriver 是我踩过的大坑

当时：

Selenium 最大噩梦。

其实不是代码。

而是：

驱动版本

特别经典错误

session not created

本质：

浏览器版本
!=
driver版本

当时每次浏览器更新

爬虫：

都可能直接报废。

这是 Selenium 时代的经典痛苦

后来 Playwright 出现后。

体验提升巨大。

Pixiv 是另一个世界

相比 Twitter。

Pixiv：

更偏：

站内作品系统

Pixiv 最大特点

其实是：

API化很明显

后来抓包后会发现

大量数据：

来自：

ajax
json

而不是：

传统 HTML。

所以后来我逐渐意识到

真正高效的爬虫：

不是：

解析HTML

而是：

抓接口

这是非常关键的认知升级

因为：

接口：

通常：

  • 更稳定
  • 更快
  • 数据更完整

HTML 页面

其实只是：

给人看的

API

才是：

真正给程序用的

于是后来我的习惯变成：

第一件事

不是看 HTML。

而是：

F12 -> Network

看什么

重点看：

  • XHR
  • Fetch

尤其：

json

接口。

很多网站真正的数据入口

其实都藏在这里。

后来又开始接触 Cookie

因为：

Pixiv：

很多内容。

不登录：

直接看不到。

所以后来：

开始研究：

Cookie复制
Session复用

那时候开始真正理解：

登录状态

本质是什么。

其实：

很多网站登录。

核心：

就是：

服务端识别Cookie

后来我开始逐渐讨厌 Selenium

因为：

它越来越像：

最后手段

能抓接口。

绝不跑浏览器。

原因非常简单

浏览器：

太重。

太慢。

太不稳定。

后来 Playwright 出现后

体验终于现代化了。

最大区别

是：

自动等待

Selenium 时代：

sleep(5)

到处都是。

Playwright：

很多操作：

自动等待页面稳定。

这其实是巨大的工程体验提升

现代爬虫最大的变化

是：

已经越来越不像：

爬网页

了。

而是：

自动化浏览器

尤其：

现代前端：

  • React
  • Vue
  • SPA

越来越多后。

页面本身

其实只是：

JavaScript应用

所以后来我的认知变化非常大

以前觉得：

爬虫 = requests

后来发现：

现代爬虫本质：

是：

“理解网站运行逻辑”。

再后来

甚至开始接触：

  • websocket
  • graphql
  • token
  • 指纹检测
  • Cloudflare

才意识到

现代反爬。

其实已经越来越接近：

攻防对抗

了。

最后总结

这类项目。

其实对我的帮助非常大。

因为：

它让我第一次真正开始：

  • 看浏览器 Network
  • 理解前后端通信
  • 理解 Cookie
  • 理解 Session
  • 理解动态网页
  • 理解接口
  • 理解自动化浏览器

很多人以为爬虫只是“采集数据”

但实际上。

它非常锻炼：

对整个Web系统的理解

而现代爬虫的发展路线

本质其实是：

从HTTP请求
逐渐走向浏览器自动化

最后一个很现实的问题

后来我越来越意识到：

很多网站。

其实：

并不欢迎高频采集

所以：

真正长期稳定的项目。

最终还是应该：

  • 控制频率
  • 尊重站点规则
  • 减少压力
  • 合理缓存

否则：

即便技术能做到。

项目也很难长期稳定运行。
