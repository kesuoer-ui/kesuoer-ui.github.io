我用 C + EasyX 开发黑白棋游戏的过程

大学课程设计时期，做过一个黑白棋（Reversi / Othello）小游戏。

当时要求：

  • 必须有图形界面
  • 需要玩家交互
  • 最好有 AI
  • 尽量体现算法设计

最后选了：

C语言 + EasyX

原因很简单：

  • SDL 太重
  • WinAPI 太折磨
  • Qt 当时不会
  • 控制台又太丑

而 EasyX：

几乎是 C/C++ 图形开发新手村

对课程设计来说刚好合适。

黑白棋为什么适合课程设计

规则简单，但具备典型博弈结构：

非常适合做 AI 博弈

核心特征：

  • 有明确状态空间
  • 有合法行动集合
  • 有局面优劣评估
  • 有可预测的未来展开

相比五子棋：

搜索树更容易爆炸

黑白棋反而更适合做出“可用 AI”。

黑白棋规则

初始布局

⚪ ⚫
⚫ ⚪

落子规则

必须满足“夹子”规则：

夹住对方棋子才能落子

示例：

⚫ ⚪ ○

在右侧落 ⚫ 后：

⚫ ⚫ ⚫

被夹住的棋子发生翻转。

结束条件

满足任一条件：

  • 双方均无合法落子
  • 棋盘填满

最终：

棋子数量多者获胜

项目结构设计

整体结构比较直观：

/project
 ├── main.c
 ├── board.c
 ├── ai.c
 ├── render.c
 ├── input.c
 └── resource/

模块划分：

  • board：棋盘状态
  • ai：AI 决策
  • render：绘制
  • input：输入处理

棋盘数据结构

最直接的实现方式：

int board[8][8];

编码定义：

0 = 空
1 = 黑
2 = 白

优点：

  • 简单
  • 直观
  • 易调试

核心逻辑：合法落子判断

黑白棋核心本质：

某个位置是否可落子

八方向扫描

翻转可能发生在 8 个方向：

dx[8] = {-1,-1,-1,0,0,1,1,1};
dy[8] = {-1,0,1,-1,1,-1,0,1};

逻辑遍历每个方向。

判断流程

以黑子为例：

起点为空格
↓
相邻必须出现白子
↓
沿方向持续延伸
↓
最终必须遇到黑子

才构成合法落子。

伪代码

for each direction:
    扫描连续敌方棋子
    如果最终遇到己方棋子:
        判定为合法

这是第一次“算法化”的部分

这里已经不再是：

UI if-else

而是：

状态空间推演

可落子提示功能

实现方式：

  • 扫描所有格子
  • 标记合法落点
  • 用 UI 提示

显示效果类似：

⚫ ⚪ ·

其中 · 为提示点。

UI 设计意义

这个功能本质上属于：

降低认知成本

避免玩家：

不知道哪里可以下

EasyX 绘制能力

例如绘制提示点：

fillcircle(x, y, r);

UI 实现中的问题

实际开发中难点不在算法，而在 UI：

  • 坐标映射
  • 输入事件
  • 屏幕刷新
  • 重绘机制

坐标映射

鼠标到棋盘：

grid_x = mouse_x / CELL_SIZE;
grid_y = mouse_y / CELL_SIZE;

重绘机制

需要完整帧刷新：

清屏
→ 画棋盘
→ 画棋子
→ 画提示

AI 设计

初始 AI 版本：

找到第一个合法点就落子

效果：

非常弱

局势评分引入

AI 改进为：

评估每一步的局面价值

搜索思想

核心逻辑：

当前步
↓
对手响应
↓
再一步
↓
递归展开

Minimax 思想

本质接近：

极大极小搜索

决策目标：

最大化自身最优结果

深度搜索（动态规划式实现）

实现方式：

固定深度递归

搜索深度影响

深度 效果
1 随机行为
2 勉强可玩
4 有压制感
6+ 性能明显下降

搜索爆炸原因

每一步都有多个分支：

分支数 ^ 深度

例如：

10^6 = 1,000,000

AI评分函数设计

棋子差值

己方 - 对方

角点权重

角点极其重要：

corner += 100;

边缘权重

边缘 > 中央

危险位置惩罚

角旁位置：

负权重

启发式评估

整体属于：

heuristic evaluation function

AI本质总结

实现逻辑是：

搜索 + 评分函数

EasyX特点

优点：

  • API 简单
  • 上手快
  • 适合课程设计

示例：

solidcircle();
setfillcolor();
line();

局限性

  • Windows 绑定
  • 性能一般
  • 生态较弱

AI优化缺失项

未实现内容：

  • Alpha-Beta 剪枝
  • 置换表
  • 开局库
  • 并行搜索

Alpha-Beta 剪枝

核心作用：

减少无效搜索分支

如果重做

更可能选择：

  • SDL2
  • Qt
  • ImGui
  • Web Canvas

项目意义总结

这个项目的核心价值不在工程规模，而在认知转变：

  • 从业务逻辑 → 状态建模
  • 从 UI 编程 → 博弈搜索
  • 从流程控制 → AI 决策结构

黑白棋是一个非常典型的：
