Claude Code 泄露源码带给我的 Agent 启发

Claude Code 泄露源码带来的真实启发

这件事最有价值的地方，其实不是“AI 又进步了”。

而是第一次能从工业级实现里看到：

Agent 不是一个模型封装，而是一个完整的软件系统。

在这之前，大部分 Agent 项目都停留在“组合 Demo”的阶段：

  • LLM + Prompt
  • LLM + Tool
  • LLM + Memory

能跑，但不稳定，也谈不上工程体系。

Claude Code 的源码让我第一次看到另一种东西：

它不是在做 Agent，而是在做一个“受控执行系统”。

1. 早期 Agent 的问题其实很一致

我之前做过不少 Agent demo，本质结构都类似：

用户输入 → LLM → Tool → 输出

或者稍微复杂一点：

LLM → 判断是否调用工具 → Tool → LLM总结

问题很快就会暴露：

（1）没有任务结构

模型直接开始做事，没有拆解过程。

结果就是：

  • 一步错全盘错
  • 多步骤任务极不稳定

（2）上下文越来越脏

所有历史都往 prompt 里堆：

  • token 爆炸
  • 早期信息污染后续决策
  • 模型越来越“迷”

（3）工具调用不可控

典型问题：

  • 参数乱填
  • 工具乱选
  • 反复调用同一个 tool
  • 死循环

本质是：没有执行边界

2. Claude Code 的核心变化：从“自由 Agent”变成“受控系统”

这是我觉得它和大部分 Agent 最大的差异。

不是能力更强，而是设计思想变了：

不再追求自由推理，而是限制模型行为空间。

可以拆成一句话：

Agent ≠ LLM 自动做事
Agent = LLM 在规则内做决策

3. 整体架构理解（不是源码细节，而是结构抽象）

我后来整理出来的结构大概是这样：

User Input
  ↓
Planner / Task Decomposer
  ↓
Context Builder
  ↓
Tool Router
  ↓
Executor
  ↓
State Manager
  ↓
Memory System
  ↓
Response Generator

重点不在“组件名字”，而在分层思想：

每一层只做一件事

而不是让 LLM “一把梭”。

4. Planner：真正的差异点之一

很多 Agent 最大的问题是：

直接让模型执行，而不是先规划。

Claude Code 的方式更接近：

  1. 先判断任务类型
  2. 再拆子任务
  3. 再决定工具链

例如：

用户说“修一下这个项目 bug”

不会直接：

read file → edit → return

而是先：

  • 项目结构分析
  • 依赖影响范围
  • 可能的错误来源
  • 修改路径规划

这一步本质上是：

把“执行”前置成“规划问题”

5. Context 管理：真正的工程难点

这部分比模型能力更重要。

大多数 Agent 失败不是因为不会做，而是：

上下文已经不可用了。

Claude Code 做了一个关键取舍：

不保留所有历史，而是重建上下文

核心思想是：

  • 不是“记录全部”
  • 而是“维护当前任务所需信息”

典型做法包括：

  • 历史摘要化
  • 任务相关信息筛选
  • 无关上下文丢弃
  • 状态重写

这点本质是 Context Engineering

后来我越来越认同一个判断：

Agent 的核心能力不在模型，而在上下文治理。

6. Tool System：不是函数调用，而是能力边界系统

很多人把 Tool 当函数：

search(query)

Claude Code 的 Tool 更像：

系统能力接口，而不是函数列表

典型能力：

  • 文件系统访问
  • shell 执行
  • git 操作
  • 网络请求
  • 代码编辑

关键点不是“能不能调用”，而是：

每个 Tool 都有明确边界

包括：

  • 能做什么
  • 不能做什么
  • 输入限制
  • 风险说明
  • 失败策略

7. 执行层：中间验证机制

一个很关键但容易忽略的点：

Tool 调用 ≠ 立即执行

在工业实现中通常会加：

  • 参数校验
  • 权限检查
  • 安全过滤
  • 执行前确认（部分场景）

原因很简单：

LLM 会犯低级错误，比如：

  • 错误路径删除文件
  • 误用 shell 命令
  • 参数拼错导致破坏性操作

所以执行链不是：

LLM → Tool → 结果

而更像：

LLM → Tool Proposal → Validation → Execution → Result

8. Diff 思维：代码修改方式的关键差异

Claude Code 在代码修改上倾向于：

最小变更原则（Diff/Patch）

而不是：

  • 整文件重写
  • 大范围生成

原因很现实：

LLM 很容易破坏上下文

比如：

  • 改动无关逻辑
  • 丢函数调用关系
  • 引入隐性 bug

Diff 的本质优势是：

  • 可控
  • 可回滚
  • 可审计

9. 状态机思想（比 Agent 本身更重要）

很多早期 Agent 是：

无状态循环调用

Claude Code 更接近：

状态驱动系统

它维护：

  • 当前任务阶段
  • 当前执行状态
  • 当前工具上下文
  • 当前目标进度

本质已经接近：

Workflow System / State Machine

10. 失败处理是工业系统核心差异

Demo 和真实系统最大的区别：

Demo 假设成功路径

工业系统默认失败

Claude Code 体现得很明显：

  • 自动重试
  • 错误分类
  • fallback策略
  • 状态恢复
  • 局部回滚

这部分其实比“模型能力”更重要。

11. Prompt 不再是提示词，而是系统配置

一个很关键的认知变化：

以前觉得 Prompt 是：

写一句话让模型更聪明

现在更接近：

控制 Agent 行为的配置系统

Claude Code 的 Prompt 结构明显是分层的：

  • system prompt（规则层）
  • tool prompt（能力层）
  • planner prompt（规划层）
  • safety prompt（约束层）

本质已经不是“写提示词”，而是：

定义 AI 行为规范。

12. 和 LangGraph / Agent 框架的关系

对比下来可以看到一个趋势：

  • LangChain：组合能力
  • LangGraph：流程控制
  • Claude Code：工程系统化实现

LangGraph 解决的是：

“怎么把 Agent 变成流程”

Claude Code 解决的是：

“流程如何在真实环境稳定运行”

13. 一个更现实的结论

这次源码带来的最大变化不是技术细节，而是认知变化：

Agent 正在从实验品变成工程系统

关键能力已经不是：

  • 会不会调用工具
  • 能不能推理

而是：

  • 能不能稳定运行
  • 能不能控制风险
  • 能不能长期维护
  • 能不能处理失败

最后一点总结

这次对 Claude Code 的理解之后，我对 Agent 的判断变成了：

未来竞争不是模型能力，而是系统工程能力。

模型只是组件之一。

真正拉开差距的是：

  • Workflow设计
  • Context治理
  • Tool体系
  • 状态控制
  • 安全与失败处理
