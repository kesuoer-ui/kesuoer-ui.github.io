# 我用 C + EasyX 开发黑白棋游戏的过程

大学课程设计时期，做过一个黑白棋（Reversi / Othello）小游戏。

当时要求：

- 必须有图形界面
- 需要玩家交互
- 最好有 AI
- 尽量体现算法设计

最后选了：

```text
C语言 + EasyX
```

原因很简单：

- SDL 太重
- WinAPI 太折磨
- Qt 当时不会
- 控制台又太丑

而 EasyX：

```text
几乎是 C/C++ 图形开发新手村
```

对课程设计来说刚好合适。

---

# 黑白棋为什么适合课程设计

规则简单，但具备典型博弈结构：

```text
非常适合做 AI 博弈
```

核心特征：

- 有明确状态空间
- 有合法行动集合
- 有局面优劣评估
- 有可预测的未来展开

相比五子棋：

```text
搜索树更容易爆炸
```

黑白棋反而更适合做出“可用 AI”。

---

# 黑白棋规则

## 初始布局

```text
⚪ ⚫
⚫ ⚪
```

---

## 落子规则

必须满足“夹子”规则：

```text
夹住对方棋子才能落子
```

示例：

```text
⚫ ⚪ ○
```

在右侧落 ⚫ 后：

```text
⚫ ⚫ ⚫
```

被夹住的棋子发生翻转。

---

## 结束条件

满足任一条件：

- 双方均无合法落子
- 棋盘填满

最终：

```text
棋子数量多者获胜
```

---

# 项目结构设计

整体结构比较直观：

```text
/project
 ├── main.c
 ├── board.c
 ├── ai.c
 ├── render.c
 ├── input.c
 └── resource/
```

模块划分：

- board：棋盘状态
- ai：AI 决策
- render：绘制
- input：输入处理

---

# 棋盘数据结构

最直接的实现方式：

```c
int board[8][8];
```

编码定义：

```text
0 = 空
1 = 黑
2 = 白
```

优点：

- 简单
- 直观
- 易调试

---

# 核心逻辑：合法落子判断

黑白棋核心本质：

```text
某个位置是否可落子
```

---

# 八方向扫描

翻转可能发生在 8 个方向：

```c
dx[8] = {-1,-1,-1,0,0,1,1,1};
dy[8] = {-1,0,1,-1,1,-1,0,1};
```

逻辑遍历每个方向。

---

# 判断流程

以黑子为例：

```text
起点为空格
↓
相邻必须出现白子
↓
沿方向持续延伸
↓
最终必须遇到黑子
```

才构成合法落子。

---

# 伪代码

```text
for each direction:
    扫描连续敌方棋子
    如果最终遇到己方棋子:
        判定为合法
```

显示效果类似：

```text

```

其中 `·` 为提示点。

---

# 搜索深度影响

| 深度 | 效果         |
| ---- | ------------ |
| 1    | 随机行为     |
| 2    | 勉强可玩     |
| 4    | 有压制感     |
| 6+   | 性能明显下降 |

---

# 搜索爆炸原因

每一步都有多个分支：

```text
分支数 ^ 深度
```

例如：

```text
10^6 = 1,000,000
```
