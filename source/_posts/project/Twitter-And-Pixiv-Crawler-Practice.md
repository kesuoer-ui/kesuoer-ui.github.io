# Twitter 与 Pixiv 爬虫实践踩坑记录

这是我真正开始深入理解：

```text
现代网站不是“网页”
而是“前端应用”
```

的一次项目实践。

当时主要目的其实很简单：

```text
自动收集同人图
```

尤其：

- Twitter
- Pixiv

这种二次元内容站。

---

# 最开始的想法其实很幼稚

一开始我以为：

```python
requests.get()
```

就结束了。

后来才发现：

> 根本不是这么回事。

---

# 当时 Twitter 还叫 Twitter

那时候：

还没改名 X。

而且：

API 也没现在这么离谱。

但：

反爬已经很严重了。

---

# 第一次真正理解“动态网页”

最开始。

我写的：

其实是：

```text
Selenium + XPath
```

传统浏览器爬虫。

---

# 当时的核心思路

---

# 1. 浏览器打开页面

```python
self.bro.get(self.link)
```

---

# 2. 下滑页面

```python
window.scrollTo()
```

---

# 3. 获取 HTML

```python
content = self.bro.page_source
```

---

# 4. XPath 提取图片

```python
tree.xpath('//img/@src')
```

---

# 5. 下载图片

```python
urllib.request.urlretrieve()
```

---

# 现在看起来很土

但：

当时其实已经能跑起来了。

---

# 我第一次意识到 Twitter 是“无限流”

因为：

Twitter：

根本不是：

```text
翻页网站
```

---

# 而是：

```text
无限下滑加载
```

---

# 所以：

传统：

```text
下一页
```

逻辑直接失效。

---

# 当时的解决方案

是：

```python
window.scrollTo()
```

不断下滑。

---

# 这里有个经典问题

很多新人第一次都会踩：

```text
滚太快
```

---

# 因为：

页面：

还没加载完成。

你已经：

```text
继续往下滚了
```

---

# 所以后来我写了：

```python
sleep(self.waittime)
```

强制等待。

---

# 现在看其实不优雅

但：

早期 Selenium 项目。

几乎：

```text
全是 sleep()
```

---

# XPath 是我第一次真正理解网页结构

当时：

Chrome F12。

基本天天开着。

---

# 我那时候开始疯狂看

例如：

```html
<div>
  <section>
    <article>
      <img />
    </article>
  </section>
```

# 然后：

```python
//body//main//section//article//img/@src
```

开始越来越长。

---

# 后来才发现

这种写法：

其实非常脆弱。

# Twitter 最大问题

# 今天还能用。
