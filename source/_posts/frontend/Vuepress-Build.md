# 我为什么开始用 VuePress 写项目文档

最开始其实没想过要做“文档站”。

当时做的是一个 Bot 项目，功能一点点往上堆，最后开始出现一些很典型的问题：

- 新用户不会用
- 群里重复问同样的问题
- 配置越来越复杂
- 功能越来越分散
- 更新内容没人知道
- 我自己隔一段时间也会忘怎么配

一开始还能靠 README 撑一下，但很快就失控了。

写到后面我基本是这种状态：

> README 变成“临时说明书”，但没人真的完整看完。

然后我才开始意识到一件事：

**文档不是附属品，是项目的一部分。**

---

# 为什么选 VuePress

当时刚接触 Vue，看到很多成熟项目的文档站：

- Vue
- Vite
- Element Plus
- Pinia

结构都很统一：

- 左侧导航
- Markdown 编写
- 自动路由生成
- 静态部署
- 代码高亮

对当时的我来说，这套东西刚好匹配需求：

- 只写 Markdown 就够了
- 不想自己造前端框架
- 需要分类、导航、版本管理
- 后面还要持续维护

VuePress 基本就是“刚好能用”。

---

# 为什么不继续用 README

后面回头看，README 最大的问题不是“写不下”，而是结构会崩。

## 1. 信息会堆成一坨

安装、配置、指令、FAQ、更新日志全混在一起，后期基本没人愿意读。

## 2. 没有层级

Bot 项目通常是多模块的：

- 群管
- 查询
- 图片
- 娱乐
- API
- 管理后台

README 很难拆开管理。

## 3. 很难维护

功能一多之后，README 会变成：

- 旧配置没删
- 新功能补在底部
- 示例过期但没人改

最后就是“看起来很全，但实际不可用”。

---

# VuePress 在当时的定位

当时的 VuePress（v1）更像一个：

> Markdown 驱动的静态文档生成器

对我这种情况比较合适的点在于：

- 文件即页面
- 目录即导航
- 不需要复杂构建流程
- 可以按模块拆文档

整体结构最后被我拆成这样：

```text
src/
├── about/
├── use/
├── group/
├── image/
├── query/
├── admin/
├── update/
└── faq/
```

每个目录单独维护一块功能说明。

---

# 当时的项目初始化

VuePress v1 + Reco 主题（当时比较常见的组合）：

```json
{
  "scripts": {
    "dev": "SET NODE_OPTIONS=--openssl-legacy-provider && vuepress dev src --open",
    "build": "SET NODE_OPTIONS=--openssl-legacy-provider && vuepress build src"
  },
  "devDependencies": {
    "vuepress": "1.9.7",
    "vuepress-theme-reco": "1.6.17"
  }
}
```

---

# Reco 主题的选择

选 Reco 其实没什么复杂原因，就是：

- 默认主题太朴素
- 想要一点“博客/产品站”的感觉
- 当时社区用的人也多

后来看，它的价值更多在于：

> 省掉了自己写首页和布局的时间。

---

# 文档结构怎么慢慢变复杂的

一开始只是写功能说明，后面逐渐变成一个完整体系。

比如：

## 插件文档结构

```text
群管插件
  - 禁言
  - 踢人
  - 全体管理

查询插件
  - 天气
  - 搜索
  - 图片

娱乐插件
  - 点歌
  - 抽卡
```

每个功能都拆成独立 Markdown 文件。

这样做之后才发现一个变化：

> 文档开始反过来约束项目结构。

---

# 更新日志的意义

后来单独做了 update 模块之后，才逐渐形成习惯：

每次改功能不再只发群公告，而是：

```md
# v1.2.0

- 新增图片功能
- 修复查询接口异常
- 优化缓存逻辑
```

当时的体感是：

- 用户不再依赖“记忆”
- 功能变化可以追溯
- 项目状态变得可见

但也踩过坑：更新写得太随意，后面回头看其实很难复现当时改了什么。

---

# VuePress 当时踩的一个环境坑

当时 Node 版本和 VuePress v1 有兼容问题，经常直接报 OpenSSL 错误。

最后用的是：

```bash
SET NODE_OPTIONS=--openssl-legacy-provider
```

现在回头看这个处理方式其实比较“临时”，但当时确实是最快能跑起来的方法。

---

# 插件系统带来的“产品化错觉”

当时 VuePress 可以很方便加各种插件：

- 动态标题
- 看板娘
- 背景特效
- 公告弹窗

一度会有种错觉：

> 好像在做一个真正的网站产品，而不是文档。

但后面也意识到一个问题：

这些东西对“文档本身”其实没什么帮助，只是视觉层面增强。

---

# VuePress v1 的问题（当时的真实感受）

后期项目一大之后问题开始明显：

- 启动变慢
- 构建时间增加
- webpack 配置老旧
- 插件兼容问题频繁

尤其是文档目录变大后，热更新体验会明显下降。

但当时没有太多替代方案，VitePress 生态还没完全成熟。

---

# 这件事对我后来的影响

回头看，这个 VuePress 项目真正带来的不是技术，而是一个转变：

从“写代码”变成“维护项目”。

开始意识到：

- 功能不是随便加的
- 结构需要提前规划
- 文档本身也是系统的一部分
- 项目会随着时间自然膨胀

也就是从这个阶段开始，我后面做任何项目都会默认：

> 先考虑“怎么维护”，再考虑“怎么实现”。

---

# 后记

现在再看 VuePress v1，其实已经有点过时了。

但它在当时的作用更像一个过渡工具：

- 帮我把散乱的说明变成结构化文档
- 逼着我开始整理项目边界
- 让“文档”和“功能”解耦

严格来说，这个项目本身并不复杂，但它是我第一次明显感觉到：

> 一个项目如果没有结构，会在后期自然崩掉。
