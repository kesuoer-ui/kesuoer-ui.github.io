## 背景

第一次接触正则表达式时，基本没有任何“工具感”，更像是在看一种陌生符号系统：

```text id="p2k8n1"
^ $ .* + ? [] () \d \w
```

直观感受就是：

> 完全不知道它在干什么。

---

## 正则的学习阶段

正则的理解一般会经历几个阶段：

### 1. 完全看不懂

```text id="a8x1p3"
无法理解语法含义
```

---

### 2. 直接复制现成表达式

```text id="c4m7v2"
网上找一个能用的 regex
```

---

### 3. 开始做局部修改

```text id="q9n2s6"
能根据需求改一小段
```

---

### 4. 能识别文本模式问题

逐渐开始意识到：

> 很多文本处理问题，本质是“模式匹配问题”。

---

## 正则的核心价值

正则真正的作用不是“写复杂表达式”，而是：

> 批量处理文本结构化问题。

常见场景包括：

- 数据清洗
- 日志分析
- 表单校验
- 文本提取
- IDE 批量替换
- CLI 文本处理

本质都是：

> 从非结构化文本中提取结构。

---

## 第一次体会正则的价值

最直观的体验来自批量处理文本时。

对比非常明显：

- 之前：手动复制、删除、修改
- 之后：一个表达式批量处理

效率差异是数量级的。

---

## 正则的本质

可以简化为一句话：

> 描述字符串规则的表达式语言。

例如：

```regex id="v7k2m4"
\d+
```

含义：

```text id="n1p9q3"
匹配一个或多个数字
```

---

## 常见基础符号

### 数字

```regex id="d1x8m2"
\d
```

---

### 字母数字下划线

```regex id="w6p3t9"
\w
```

---

### 空白字符

```regex id="s3k7n5"
\s
```

---

### 任意字符

```regex id="t9q1v4"
.
```

---

### 重复控制

```regex id="h2m8c6"
*  +  ?
```

含义分别是：

```text id="z4n7p2"
0次或多次
1次或多次
0次或1次
```

---

## 边界符号

### 开头

```regex id="b8x3q1"
^
```

### 结尾

```regex id="k5m9v7"
$
```

含义：

- `^` 表示字符串开始
- `$` 表示字符串结束

---

## 一个典型例子：手机号校验

```regex id="p7n2x9"
^1\d{10}$
```

含义：

```text id="r3m8v1"
以1开头 + 10位数字 + 总长度11位
```

---

## 表单校验的常见场景

正则最常见的入口通常来自：

- 手机号
- 邮箱
- 密码规则
- 身份证
- URL 校验

这些场景的特点是：

> 规则明确 + 输入格式固定。

---

## 超长正则的问题

实际开发中经常会遇到这种情况：

```regex id="q1v8m3"
((([a-zA-Z0-9...
```

问题不在于“能不能用”，而在于：

- 可读性极差
- 不可维护
- 难以调试
- 团队协作困难

---

## 更合理的使用方式

后来更倾向于：

> 能拆逻辑就拆逻辑。

避免把复杂业务逻辑压缩进一个表达式。

---

## 正则的典型实战场景

### 1. 数据清洗

#### 提取数字

```regex id="n8x4m2"
\d+
```

---

#### 去除 HTML 标签

```regex id="c6p1t9"
<[^>]+>
```

---

#### 清理多余空行

```regex id="v2m7q3"
\n+
```

---

#### 提取 URL

```regex id="s9k3n6"
https?://[^\s]+
```

---

## 2. IDE 批量替换

典型使用方式是：

> 利用 regex 做全局结构替换

例如：

```text id="d3n8m1"
log.xxx -> logger.xxx
```

---

## 3. 日志分析

日志通常是结构化文本，例如：

```text id="h7m2v8"
[ERROR] user_id=123
```

提取字段：

```regex id="p2q8n4"
user_id=(\d+)
```

---

## 4. 爬虫中的文本提取

虽然有更专业解析工具，但 regex 在简单场景仍然常用。

例如提取图片：

```regex id="t4n9m2"
src="([^"]+)"
```

---

## 不同语言中的 regex

### Python

```python id="x3m7q1"
import re

re.findall(r"\d+", text)
```

特点是：

- 语法简洁
- 使用频繁

---

### JavaScript

```javascript id="k8p2v6"
text.match(/\d+/g);
```

---

### Java

```java id="v6m1n9"
Pattern.compile("\\d+")
```

特点：

> 转义较繁琐

---

### Go

```go id="q9m4x2"
regexp.MustCompile(`\d+`)
```

特点：

> raw string 相对友好

---

### C#

```csharp id="n7v3m5"
Regex.Match(text, @"\d+")
```

---

## regex 的局限性

正则并不适合所有问题。

不适合：

- 嵌套结构
- 递归语义
- 复杂上下文依赖

例如解析 HTML：

> 并不是正则的合理使用场景。

---

## 正则的合理边界

更合适的范围是：

> 规则明确的字符串匹配与提取

而不是：

> 复杂语义结构解析

---

## 工程中的实际位置

在实际开发中，regex 更多出现在：

- 编辑器（VSCode / IDEA）
- Linux 工具（grep / sed）
- 日志处理脚本
- 配置处理脚本

本质上属于：

> 文本处理工具链的一部分

---

## 总结

正则表达式本质上不是“复杂技巧”，而是：

> 一种高效处理文本规则的工具。

它的价值不在于表达式本身的复杂度，而在于：

- 提升批量处理能力
- 降低重复劳动
- 提高文本处理效率

从工程角度看，它更像是：

> 文本处理层的基础工具能力，而不是高级算法。
