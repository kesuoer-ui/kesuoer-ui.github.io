# 关系型与非关系型数据库的理解整理

数据库这块，我一开始其实是没什么体系感的。

最早写后端的时候，基本就是：

- CRUD
- 查表
- 写接口
- 能跑就行

直到系统稍微复杂一点之后，问题就开始变了：

- 为什么这个 SQL 会慢
- 为什么加了索引还是慢
- 为什么线上偶尔卡死
- 为什么数据不一致
- 为什么一加并发就出问题

那时候才慢慢意识到：

数据库不是“存数据的地方”，更像是整个系统的底层调度器。

---

# 一、关系型数据库（RDBMS）

最早接触的一类就是关系型数据库。

常见的基本就是：

- MySQL
- PostgreSQL
- Oracle
- SQL Server
- SQLite

当时我的理解很简单：

> 表 + SQL + 能查数据 = 数据库

后来做项目多了才发现，关系型数据库真正核心是“关系”。

比如电商系统：

- 用户
- 商品
- 订单
- 支付

这些东西单独看都简单，但一旦连起来，就全是关系。

---

## MySQL（最常用的一类）

我第一个正式用在项目里的数据库就是 MySQL。

用下来最大的感受其实不是“强”，而是：

> 足够稳，够通用，出了问题也好查。

基本 Web 项目都绕不开它。

---

## InnoDB 和 MyISAM

我当时踩坑比较深的一点是存储引擎。

- MyISAM：早期用过，后来基本放弃
- InnoDB：现在默认用

区别当时我其实是踩出来的：

MyISAM 写入稍微多一点就容易出问题，而且事务基本没法用。

InnoDB 用起来就正常很多：

- 能回滚
- 支持事务
- 崩了还能恢复

---

## 事务这块（ACID）

事务这块我第一次真正有感知，是在做转账逻辑的时候。

大概就是这种逻辑：

```sql
BEGIN;

UPDATE user SET money = money - 100 WHERE id = 1;
UPDATE user SET money = money + 100 WHERE id = 2;

COMMIT;
```

当时踩过一次问题是：

中间服务挂了，钱扣了但没加上。

后来才知道数据库会帮你回滚，这就是事务的意义。

ACID 当时是背的，后来是靠事故理解的：

- 原子性：要么都成功，要么都失败
- 一致性：数据不能乱
- 隔离性：互不影响
- 持久性：提交了就不会丢

---

## 隔离级别

这个其实一开始我也只是背概念。

真正有感觉是在高并发场景里。

四个级别：

- 读未提交
- 读已提交
- 可重复读（MySQL 默认）
- 串行化

后来才慢慢理解为什么默认是 RR：

性能和一致性之间的一个折中。

---

## MVCC（后来才真正理解）

MVCC 我最早是“知道这个词”，但没感觉。

后来在并发读写冲突里才慢慢理解：

数据库其实是靠“版本”在解决并发读的。

简单说就是：

> 读的不是当前数据，而是某个时间点的快照。

这样才做到：

- 读不阻塞写
- 写不阻塞读

---

## 锁（真正踩坑的地方）

锁这个东西我是在线上问题里学的。

典型问题就是死锁。

比如两个事务：

- A 锁 user 再等 order
- B 锁 order 再等 user

直接卡死。

后来总结下来最有效的办法其实很朴素：

- 加锁顺序统一
- 事务尽量短
- 少做大范围更新

---

## 索引（性能问题核心）

性能问题 80% 其实都和索引有关。

我当时最大的误区是：

> 慢就加索引。

后来踩坑才知道不是这么简单。

几个典型坑：

- like '%xxx' 直接失效
- 对字段做函数会失效
- 类型不一致会失效

最核心的理解其实是：

> 索引不是加了就用，而是查询方式必须匹配它的结构。

---

## B+Tree

InnoDB 默认用 B+Tree。

我当时理解比较简单：

- 树越矮越快
- 磁盘 IO 越少越好
- 范围查询很适合

后面看执行计划（EXPLAIN）才慢慢有感觉。

---

# 二、PostgreSQL

PostgreSQL 是后面接触的。

当时换的原因很现实：

> MySQL 在一些复杂查询和结构化能力上开始不够用了。

PG 给我最直观的感觉是：

- SQL 更规范
- JSON 支持更自然
- 扩展能力强
- 复杂查询更舒服

尤其 JSONB，我后来在一些配置类数据里用得很多。

---

# 三、非关系型数据库（NoSQL）

后面系统复杂起来之后，开始接触 NoSQL。

最常见的几个：

- Redis
- MongoDB

---

## Redis（我用得最多的 NoSQL）

Redis 一开始我理解就是缓存。

后来用多了才发现它其实更像：

> 内存数据结构服务。

常见用法：

- 缓存
- Token
- 限流
- 排行榜
- 分布式锁

---

### 为什么快

这个其实是踩坑之后慢慢理解的：

- 内存操作
- 单线程避免锁竞争
- IO 多路复用

不是“数据库快”，而是压根不走传统数据库那套路径。

---

### 我踩过的几个坑

缓存相关问题基本都遇到过：

- 缓存穿透（查不存在的数据）
- 缓存击穿（热点失效瞬间打爆数据库）
- 缓存雪崩（大量 key 同时过期）

后来处理方式基本就几种：

- 空值缓存
- 随机过期时间
- 热点保护

---

## MongoDB

MongoDB 用得相对少一些。

主要在一些结构不稳定的数据里用过：

比如：

- 配置
- 日志
- 用户扩展字段
- 一些 JSON 结构数据

它的特点很直接：

> 不强约束结构，灵活，但一致性弱一些。

---

# 四、数据库之间的分工理解

后来我对数据库的理解变成了比较工程化的一种拆分：

---

## 关系型数据库

适合：

- 强一致性
- 核心业务数据
- 结构稳定

比如：

- 订单
- 用户
- 资金

---

## Redis

适合：

- 快速访问
- 缓存
- 状态类数据
- 临时数据

---

## MongoDB / JSON 类数据库

适合：

- 结构不固定的数据
- 配置类数据
- 日志类数据

---

# 五、一些后期才理解的东西

## 1. 数据库慢不一定是数据库的问题

很多时候是：

- 索引设计问题
- SQL 写法问题
- 数据量设计问题

---

## 2. ORM 不是银弹

我早期很依赖 ORM，后来踩坑后变成：

- 简单查询用 ORM
- 复杂查询直接 SQL

---

## 3. 系统复杂之后，数据库只是其中一环

后面做系统会发现：

真正复杂的不是 SQL，而是：

- 缓存一致性
- 并发控制
- 数据流设计
- 异步链路

---

# 六、最后的理解变化

回头看数据库这条线，其实变化挺明显的：

一开始是：

> 会写 SQL 就够了

后来变成：

> 能跑就行，但开始关注性能

再后来变成：

> 数据结构、并发、缓存、分布式一起看

数据库本身没有变复杂，是系统把它放到了更复杂的位置上。
