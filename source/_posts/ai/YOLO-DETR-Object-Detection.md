# YOLO 与 DETR：目标检测模型理解整理

本科做毕设的时候，真正让我感觉“深度学习开始进入工程领域”而不只是调包调用的，就是目标检测。

尤其是遥感方向。

因为遥感图像和普通互联网图片完全不是一个世界。

你会开始接触：

- 超大分辨率
- 小目标密集分布
- 船舶方向旋转
- 多尺度
- 超长宽比
- 推理速度
- 显存限制
- 数据集标注质量
- 数据增强
- backbone
- neck
- feature map
- transformer

甚至第一次开始认真看论文、改网络结构、做模型融合、研究 loss 和指标。

这一阶段的技术跨度其实非常大。

---

# 目标检测到底在解决什么问题？

很多人第一次接触 YOLO 时，其实根本没意识到：

> “目标检测” 和 “图像分类” 根本不是一回事。

---

## 图像分类

分类只回答：

> “图里有什么？”

例如：

```text
这是一只猫
这是一艘船
这是飞机
```

````

模型输出一个类别即可。

---

## 目标检测

检测需要回答：

```text
图里有什么？
它在哪里？
有几个？
```

于是输出会变成：

```json
[
  {
    "class": "ship",
    "x": 123,
    "y": 456,
    "w": 78,
    "h": 45
  }
]
```

也就是：

- 类别
- 坐标
- 宽高

一起预测。

---

# 目标检测为什么难？

因为它本质上属于：

> “结构化视觉理解”

不是简单分类。

---

# 一个经典问题：同一张图里有很多目标

例如遥感图：

- 船
- 港口 -建筑
- 云层
- 海面噪声

而且：

- 尺寸差异极大
- 方向不固定
- 密集重叠
- 分辨率很高

这时检测难度会急剧上升。

---

# YOLO 系列为什么能统治这么多年

YOLO 本质上是：

> One Stage Detector

即：

```text
一次前向传播
直接输出检测结果
```

不走：

```text
RPN -> Proposal -> 分类
```

这种两阶段路线。

因此：

- 快
- 简洁
- 工业部署方便

所以工业界非常喜欢。

---

# YOLO 的核心思路

YOLO 最经典的思想其实非常暴力：

> “把图像切成网格，每个格子负责预测目标”

例如：

```text
13x13
26x26
52x52
```

每个格子：

- 预测类别
- 预测位置
- 预测置信度

---

# Anchor 是什么？

早期 YOLO 大量使用 Anchor。

这是很多新人第一次接触时最迷惑的东西。

---

## 本质

Anchor 可以理解为：

> “预设的候选框模板”

例如：

```text
小方框
长方框
细长框
```

模型不是从零预测。

而是：

```text
在 Anchor 基础上微调
```

这样更稳定。

---

# Anchor 为什么后来逐渐被嫌弃？

因为它问题很多。

---

## 1. 超参数地狱

你需要设置：

- anchor数量
- anchor比例
- anchor尺寸

不同数据集都不同。

---

## 2. 小目标容易炸

遥感图大量小目标。

Anchor 很难完全覆盖。

---

## 3. 正负样本分配复杂

训练时需要：

```text
哪个anchor负责哪个目标？
```

规则非常复杂。

---

# 所以后来开始流行 Anchor-Free

即：

> 不再预设框。

模型直接预测。

例如：

- FCOS
- YOLOX
- RT-DETR

都大量引入了 Anchor-Free 思想。

---

# YOLO 真正强的地方：工业化

很多人会误以为：

> “YOLO 强是因为精度最高”

其实不是。

YOLO 最大优势一直是：

- 推理快
- 部署简单
- 生态成熟
- TensorRT支持好
- ONNX支持好
- 社区庞大

工业界非常吃这一套。

---

# YOLO 的典型结构

现代 YOLO 一般分：

```text
Backbone
Neck
Head
```

---

# Backbone

负责：

> 特征提取

类似：

```text
ResNet
CSPDarkNet
ConvNeXt
```

---

# Neck

负责：

> 多尺度特征融合

典型：

```text
FPN
PAN
BiFPN
```

因为：

- 小目标
- 大目标

需要不同层级特征。

---

# Head

负责：

- 分类
- 回归
- 置信度

最终输出检测框。

---

# NMS 是什么？

这是 YOLO 系非常经典的一步。

---

## 问题

模型会预测大量重叠框：

```text
同一艘船
预测了20个框
```

于是需要：

> Non-Maximum Suppression

保留置信度最高那个。

---

# 但 NMS 一直是个“补丁式方案”

因为它本质不是神经网络的一部分。

而是后处理。

于是：

- 阈值难调
- 密集目标容易误杀
- 小目标容易被压掉

---

# DETR 为什么会出现？

因为 Transformer 开始席卷 CV。

Facebook 提出了：

# DETR

Detection Transformer。

它想解决：

> “目标检测为什么一定需要 NMS？”

---

# DETR 的核心思想

DETR 直接：

```text
输入图像
输出固定数量目标
```

甚至：

- 不需要 Anchor
- 不需要 NMS

---

# 它怎么做到的？

核心在：

# Object Query

可以理解成：

```text
一组“检测槽位”
```

例如：

```text
100个 query
```

每个 query：

```text
负责寻找一个目标
```

---

# Hungarian Matching

这是 DETR 非常关键的一步。

训练时：

```text
预测框
真实框
```

需要“一一匹配”。

于是用了：

# 匈牙利匹配算法

避免：

```text
多个query抢同一个目标
```

---

# DETR 的问题

虽然很优雅。

但早期 DETR 有几个巨大问题。

---

## 1. 收敛巨慢

训练非常痛苦。

有时：

```text
训练几百epoch
```

才正常。

---

## 2. 小目标差

Transformer 对局部细节不敏感。

遥感尤其明显。

---

## 3. 推理不够快

工业部署不够舒服。

---

# RT-DETR 的意义

后来百度出了：

# RT-DETR

本质是：

> “让 DETR 真正工业化”

---

# RT-DETR 做了什么？

核心：

- 提升实时性
- 优化推理
- 保留Transformer优势

它大量优化：

- 特征提取
- Query交互
- Decoder结构

---

# 我当时为什么关注 RT-DETR

因为毕设是：

# 遥感旋转框目标检测

即：

```text
YOLOv8 OBB
```

这时会发现：

---

# 遥感图像真的太恶心了 🤦

普通目标检测：

```text
猫狗汽车
```

目标方向大体正常。

---

# 但遥感图像：

船可能：

- 倾斜
- 横放
- 竖放
- 密集停靠

普通水平框：

```text
HBB
```

会非常不准。

---

# 所以开始出现 OBB

即：

# Oriented Bounding Box

旋转框。

---

# HBB vs OBB

普通框：

```text
[x, y, w, h]
```

旋转框：

```text
[x, y, w, h, angle]
```

多了角度。

---

# 但问题来了

角度预测极其容易炸。

---

# 为什么？

因为：

```text
179°
-179°
```

实际上方向很接近。

但数值差巨大。

这就是：

# 角度周期性问题

---

# OBB 训练为什么难

还有：

- 长宽交换问题
- 边界跳变
- loss不稳定
- 标注误差
- 密集重叠

这些都很折磨。

---

# 数据增强在遥感里非常重要

我后面其实大量时间都花在：

# 数据增强

而不是网络结构。

---

# 常见增强

```text
Mosaic
MixUp
HSV
随机旋转
随机裁剪
尺度缩放
```

---

# 为什么增强这么重要？

因为遥感数据：

> 太少了。

尤其高质量标注集。

---

# 小目标问题

遥感最经典的问题：

# 小目标。

例如：

```text
4000x4000 图
船只有十几像素
```

普通 backbone 很容易：

```text
下采样后直接没了
```

---

# 所以后来会大量关注：

- 高分辨率特征层
- FPN
- PAN
- 特征融合

---

# 我当时踩过的几个坑

---

# 1. 只会堆模型

一开始总觉得：

```text
换个更大的backbone
```

精度就会高。

后来发现：

> 数据集问题远大于模型问题。

---

# 2. 遥感数据标注质量非常影响结果

很多框：

- 歪
- 偏
- 漏标

模型会直接学坏。

---

# 3. Batch Size 非常影响训练稳定性

尤其 Transformer。

小 batch：

- loss乱飞
- 不稳定

但显存又不够。

---

# 4. 小目标很吃输入分辨率

输入太小：

```text
船直接糊掉
```

但分辨率高：

```text
显存爆炸
```

---

# 5. OBB 的可视化调试非常重要

有时 loss 看着正常。

结果：

```text
框方向全反了
```

所以必须：

> 实时看预测图。

---

# 我后来最大的理解变化

以前会觉得：

> “AI 就是调模型”

后来发现不是。

真正占时间的是：

- 数据
- 标注
- 增强
- 推理
- 显存
- 部署
- 训练稳定性

甚至：

```text
如何不炸显存
```

都属于工程能力。

---

# 目标检测已经越来越工程化

现在很多比赛。

拼的已经不是：

```text
谁发明了新网络
```

而是：

- 数据处理
- 训练技巧
- pipeline
- 多模型融合
- 推理优化

---

# 现在再回头看 YOLO 和 DETR

我会觉得：

---

# YOLO

更像：

> 工业工程派

特点：

- 快
- 稳
- 好部署
- 生态成熟

---

# DETR

更像：

> 学术结构派

特点：

- 优雅
- Transformer统一化
- End-to-End

但工业化花了很久。

---

# 最后

目标检测其实是很多人第一次真正接触：

- 深度学习工程
- GPU资源
- 训练技巧
- 数据集
- benchmark
- 推理部署

的开始。

也是第一次发现：

> “模型结构只是整个系统中的一部分”

真正困难的，往往是：

- 数据
- 工程
- 算力
- 调试
- 部署
- 实际场景适配

而不是单纯换一个论文里的网络名字。
````
