我踩过的代理、转发与远程访问坑

发表于2021-11-03|更新于2026-05-11|技术
|浏览量:

背景

学会基础代理之后，折腾的方向很自然地变成了另一件事：

让不同设备之间可以互相访问。

从“能上网”，变成“能互相连”。

于是开始接触一堆工具和概念：

  • frp
  • gost
  • SOCKS5
  • Tailscale
  • Proxifier
  • 内网穿透
  • 远程访问

问题也随之变得更复杂。

公网 IP 的真实问题

一开始有一个很朴素的误解：

只要能上网，就能被访问

后来才发现这是错的。

现实情况通常是：

  • 没有公网 IP
  • NAT 嵌套 NAT
  • 运营商级 NAT（CGNAT）

结构变成：

id
外部 -> 运营商 NAT -> 家庭路由 -> 设备

结果是：

- 你可以访问外网
- 外网无法直接访问你

---

## frp：第一次接触内网穿透

frp 是第一次真正让“远程访问本地服务”变得可行的工具。

基本结构：

```text id="frp1"
内网机器 -> frpc -> 公网服务器 -> frps -> 外部访问
```

核心思想是：

> 主动向公网建立连接，而不是等待被访问。

---

## frp 带来的第一个理解

以前认为“访问 = 直接连”。

frp 让人意识到：

> 网络连接可以是反向建立的。

---

## frp 的几个典型问题

### 1. TCP ≠ 全部协议

早期配置经常是：

```ini id="frp2"
type = tcp
```

默认理解是：

> 什么都能转

但实际情况是：

- HTTP / HTTPS / WebSocket 行为不同
- UDP 需要单独处理
- 应用层协议影响转发结果

---

### 2. WebSocket 转发问题

典型现象：

```text id="ws1"
HTTP 正常
WebSocket 失败
```

后来才意识到是反代层缺少升级支持。

---

### 3. HTTPS 与证书问题

后期逐渐变成：

- mixed content
- ws / wss 区分
- 证书链
- SNI 匹配

网络问题开始变成“配置组合问题”。

---

## gost：更灵活的转发拼装工具

gost 更像一个“网络组合器”。

可以做很多协议转换：

```text id="gost1"
socks5 -> http
tcp -> websocket
udp -> tunnel
```

核心特点是：

> 协议可以自由拼接

---

## gost 的一个典型认知冲击

可以多层嵌套：

```text id="gost2"
本地 -> socks5 -> http代理 -> 远程节点
```

第一次看到时更像：

> 网络层套娃结构

---

## gost 常见问题

### 1. 配置错误频发

常见报错：

```text id="gost3"
invalid character
```

本质原因多数是：

- YAML 缩进
- 格式解析问题

---

### 2. URL 与认证格式问题

常见情况：

```text id="gost4"
http://user:pass@ip:port
```

但不同模式下解析规则不同，容易出现：

- 编码错误
- 特殊字符解析失败
- 参数不一致

---

## Proxifier：进程级代理控制

Proxifier 解决的是一个很现实的问题：

> 很多程序不支持代理配置

它的方式是：

> 强制接管进程网络流量

---

## Proxifier 的能力本质

可以理解为：

- 进程级 SOCKS5 转发
- 强制流量重定向

适用场景：

- 游戏
- 闭源软件
- 无代理设置客户端

---

## Proxifier 的典型问题

### DNS 仍然是核心坑点

常见情况：

```text id="dnsx1"
IP 可以通
域名不通
```

原因通常是：

> DNS 仍然走本地解析路径

---

## Tailscale：现代组网体验

Tailscale 是一个完全不同方向的体验。

安装之后的第一感觉是：

> 设备之间自动变成局域网

---

## Tailscale 的连接模型

```text id="ts1"
设备A ↔ 设备B ↔ 设备C
```

无需手动配置复杂路由。

---

## Tailscale 的底层认知

后来才知道其核心是：

- WireGuard
- 点对点加密连接
- NAT 穿透
- 中继（DERP）

---

## Tailscale 的限制

### 1. NAT 穿透并不总是成功

在复杂网络环境中：

- 可能无法直连
- 会走中继节点

结果是：

> 延迟明显上升

---

### 2. 手机网络不稳定

移动端常见问题：

- 后台限制
- 网络切换
- 休眠断连

表现为：

```text id="ts2"
连接间歇性中断
```

---

## 手机代理的复杂性

手机环境比桌面更复杂：

- App 网络栈不同
- QUIC 协议存在
- DNS 行为不统一
- 部分应用绕过系统代理

---

## 一个典型现象

即使系统开启代理：

```text id="mobile1"
仍然直连
```

原因可能是：

- 应用内置 DNS
- 使用 HTTP/3 (QUIC)
- 自定义网络栈

---

## SOCKS5 的长期地位

在大量工具组合之后，最终会回到一个事实：

> SOCKS5 是最通用的中间层

因为它：

- 不依赖协议类型
- 兼容 TCP / UDP
- 适配范围最广

---

## 后来的一个统一认知

随着工具越来越多，逐渐形成一个简单结论：

很多问题不再是“工具问题”，而是：

- NAT 结构
- DNS 行为
- 路由路径
- 协议兼容性
- 系统网络栈差异

---

## 小结

这一阶段真正的变化不是工具数量增加，而是理解方式发生了变化：

从“怎么配置工具”，逐渐变成：

> 网络流量是如何在不同层之间被传递和重定向的
